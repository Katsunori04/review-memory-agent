# Review Memory Agent コンセプト

## 1. プロダクト名

**Review Memory Agent**

## 2. 一言コンセプト

**AIが一次レビューし、レビュワーの判断を次回以降のレビュー品質につなげるAIエージェント**

別案：

**レビューするたびに、組織のレビュー判断が育つAI**

---

## 3. プロダクト概要

Review Memory Agentは、提案書・稟議書・設計書・システム構成図などのレビュー業務を支援するプロダクトである。

AIが資料に対して一次レビューを行い、基本的な指摘、リスク、修正案、確認事項を提示する。レビュワーはその結果を確認し、必要に応じて修正・承認する。

修正された内容は、単なる履歴として保存するのではなく、次回以降のレビューで再利用できる「レビュー方針カード」として構造化し、ベクトルストアに保存する。

次に類似資料をレビューするときは、過去のレビュー方針を参照し、組織固有の判断基準を踏まえたレビューを行う。

---

## 4. Why：なぜ必要か

### 4.1 解決したい本質課題

**レビュー判断が属人化し、過去レビューが再利用されていない。**

多くの組織では、提案書、稟議書、設計書、構成図などのレビューで、同じような指摘が繰り返されている。

例：

- 効果が定性的で、判断材料として弱い
- 費用対効果が示されていない
- セキュリティ観点が不足している
- 運用責任者が不明
- 顧客課題との対応関係が弱い
- 例外処理や非機能要件が不足している
- システム構成図に認証・権限・監視の観点が不足している

しかし、これらのレビュー判断は、メール、Teams、コメント、口頭指摘、個人メモなどに散らばり、次回以降のレビューに十分活用されていない。

その結果、レビュー担当者は毎回ゼロから確認し、資料作成者は何を直せば通るのか分からない。組織としてもレビュー品質が蓄積されにくい。

### 4.2 現状の課題

#### レビュー担当者側の課題

- 毎回ゼロから資料を確認している
- 過去に似たレビューをしたのに、その指摘を探せない
- レビュー観点がベテランや特定担当者の頭の中にある
- 同じ指摘を何度も書いている
- 忙しさや担当者によってレビュー品質にばらつきが出る
- 若手や別担当者にレビュー観点を引き継ぎにくい

#### レビュイー側の課題

- 何を直せばレビューを通るのか分からない
- レビュー観点が事前に見えない
- 人によって指摘内容が変わる
- 提出後に大きな差し戻しが発生する
- 指摘の背景や理由が分かりにくい
- レビュー前にセルフチェックできない

#### 組織側の課題

- レビュー品質が人に依存する
- ベテランレビュワーに負荷が集中する
- 過去レビューがナレッジとして残らない
- 同じ差し戻しが繰り返される
- レビューしても組織として賢くならない

### 4.3 このプロダクトが目指す状態

Review Memory Agentが目指すのは、AIにレビューを丸投げすることではない。

目指すのは、**人間のレビュー判断をAIが再利用可能な形に変換し、次回以降のレビューに活かすこと**である。

レビューのたびに、次の循環を作る。

1. AIが一次レビューする
2. レビュワーが判断する
3. 判断内容がレビュー方針として蓄積される
4. 次回以降のAIレビューに反映される

これにより、レビュー業務を単発の作業ではなく、組織学習のプロセスに変える。

## 5. What：何を提供するか

### 5.1 提供するプロダクト

Review Memory Agentは、以下の機能を提供する。

- 資料のAI一次レビュー
- 過去レビュー方針の検索
- レビュー結果の修正・承認
- レビュワー判断の学習カード化
- 学習カードのベクトルストア保存
- 類似資料レビュー時の過去方針適用
- レビュー品質の継続的改善

### 5.2 主なレビュー対象

初期スコープでは、以下を対象とする。

| レビュー対象 | 内容 |
| --- | --- |
| 顧客提案書 | 顧客課題、提案価値、効果、費用対効果、導入リスクを確認 |
| 社内稟議書 | 目的、投資対効果、代替案、リスク、承認者観点を確認 |
| 設計書 | 要件整合性、例外処理、非機能要件、テスト観点を確認 |
| システム構成図 | 認証、権限、監視、データ流通、責任分界を確認 |

将来的には、以下にも拡張する。

| 拡張対象 | 内容 |
| --- | --- |
| 法務資料 | 契約・規約・責任範囲・個人情報に関する確認観点を抽出 |
| ServiceNow / 社内QA | 過去の質問・回答・社内見解を取り込み、レビュー判断に活用 |
| FAQ / ナレッジベース | 組織内に蓄積された公式見解をレビューに反映 |

### 5.3 中核機能

#### 1. AI一次レビュー

レビュイーが資料をアップロードすると、AIが基本的なレビューを行う。

主な出力：

- 総合判定
- レビュー通過可能性スコア
- 主要指摘
- リスク
- 修正案
- 不足情報
- レビュワーに確認すべき点

#### 2. 類似レビュー検索

過去に保存されたレビュー方針カードを検索し、今回の資料に関連する判断基準を取得する。

主な検索対象：

- 類似資料
- 類似指摘
- 過去のレビューコメント
- レビュワーが承認したレビュー方針
- 資料タイプ別のレビュー観点

#### 3. レビュワー修正・承認

レビュワーはAIのレビュー結果を確認し、以下の操作を行う。

- 指摘を採用する
- 指摘を修正する
- 指摘を却下する
- 追加指摘を入れる
- 最終レビューとして承認する
- 今回の判断を次回以降も使うか選択する

#### 4. レビュー方針カード生成

レビュワーが修正した内容から、AIが再利用可能なレビュー方針カードを生成する。

例：

```text
タイトル：
業務効率化提案における効果説明のレビュー方針

適用対象：
顧客提案書、社内稟議書、PoC企画書

判断基準：
効果説明では、対象人数・利用頻度・1回あたり削減時間を用いて、月間または年間の削減効果を示すことが望ましい。

推奨レビューコメント：
効果を判断しやすくするため、対象人数・月間利用回数・1回あたり削減時間を追記してください。
現時点で試算が難しい場合は、PoCで取得する指標と測定方法を記載してください。

適用条件：
業務効率化を目的とした提案で、効果が定性的にしか記載されていない場合

除外条件：
初期アイディア段階、ブレスト資料、定量効果を求めない資料
```

#### 5. 学習カード承認

AIが生成したレビュー方針カードは、レビュワーが確認する。

選択できる操作：

- 保存する
- 修正して保存する
- 今回限りにする
- 保存しない

AIが勝手に学習するのではなく、**人間が承認した判断だけを次回以降に利用する**。

#### 6. 次回レビューへの反映

類似資料がアップロードされた際、AIは過去のレビュー方針カードを参照し、今回の資料に合わせてレビューする。

出力例：

```text
過去の承認済みレビュー方針に基づき、今回も「効果説明の定量化」と「運用責任者の明記」を重点確認しました。
```

- 修正すべき理由が分かる
- 資料品質を事前に高められる

#### レビュワーにとっての価値

- 基本的な指摘をAIが下書きしてくれる
- 過去レビューを探す手間が減る
- 同じ指摘を何度も書かなくてよい
- 自分の判断を次回以降に残せる
- レビュー観点を若手や別担当者に引き継ぎやすくなる

#### マネージャー・組織にとっての価値

組織として、レビュー品質を標準化できる。

得られる価値：

- レビュー観点の属人化を減らせる
- 過去レビューを組織資産として活用できる
- レビュー品質のばらつきを抑えられる
- ベテランレビュワーへの負荷集中を緩和できる
- レビューするほど組織の判断基準が育つ

## 7. When：いつ使うか

### 7.1 レビュー前

レビュイーが資料を作成した直後に使う。

目的は、レビュワーに出す前のセルフチェックである。

利用例：

- 提案書を上司に見せる前
- 稟議書を承認ルートに回す前
- 設計書をレビュー会に出す前
- 構成図を情シスに確認する前
- 法務資料を正式確認に出す前

### 7.2 レビュー中

レビュワーが資料を確認する際に使う。

目的は、AIレビューをたたき台にして、レビュー品質とスピードを上げることである。

利用例：

- AIの指摘を確認するΩ
- 不足している指摘を追加する
- 不要な指摘を削除する
- コメントの表現を修正する
- 最終レビューとして承認する

### 7.3 レビュー後

レビュー完了後に使う。

目的は、レビュワーの判断を次回以降に活かすことである。

利用例：

- 修正内容からレビュー方針カードを生成する
- 次回以降も使う判断基準を保存する
- 今回限りの判断として保存しない
- 類似レビューに備えてナレッジ化する

### 7.4 次回の類似レビュー時

過去に保存されたレビュー方針を、次回の資料レビューに使う。

利用例：

- 似た提案書に対して、過去の指摘観点を適用する
- 似た稟議書に対して、過去の承認観点を適用する
- 似た設計書に対して、過去の技術レビュー観点を適用する
- 同じような構成図に対して、過去のセキュリティ指摘を適用する

## 8. How：どのように実現するか

### 8.1 基本フロー

```text
1. レビュイーが資料をアップロード
2. AIが一次レビューを実施
3. 過去のレビュー方針をベクトル検索
4. AIが過去方針を踏まえてレビュー結果を生成
5. レビュイーが必要に応じて事前修正
6. レビュワーにレビュー依頼
7. レビュワーがAIレビュー結果を確認・修正
8. レビュワーが最終レビューを承認
9. AIが修正差分からレビュー方針カードを生成
10. レビュワーが保存可否を判断
11. 承認済みカードをベクトルストアに保存
12. 次回以降の類似レビューで参照
```

### 8.2 Human in the Loop の考え方

このプロダクトでは、AIが最終判断を行わない。AIの役割は、レビュー作業を支援することである。

最終判断はレビュワーが行い、その判断を次回以降に活用する。

```text
AIがレビューする
↓
レビュワーが判断する
↓
判断をレビュー方針化する
↓
次回のAIレビューに反映する
```

これにより、AIが独断で学習するのではなく、人間が承認した判断だけが組織のレビュー資産として蓄積される。

### 8.3 「賢くなる」の実現方法

MVPでは、AIモデル自体を都度ファインチューニングするのではなく、レビュー方針カードをベクトルストアに保存し、RAGで参照する。

```text
× AIモデルを毎回再学習する
○ 人間が承認したレビュー方針を検索可能なナレッジとして保存する
```

この方式により、以下を実現する。

- 実装しやすい
- 透明性が高い
- 人間が承認した方針だけを利用できる
- 適用した過去レビュー方針を画面に表示できる
- 誤った判断基準を削除・無効化できる

## 9. 代表的なアウトプット

### 9.1 AI一次レビュー結果

```text
総合判定：
要修正

レビュー通過可能性：
62 / 100

主な不足：
1. 効果説明が定性的
2. 運用責任者が不明
3. セキュリティ説明が不足
4. 導入後の評価方法が未記載

推奨アクション：
対象人数、月間利用回数、1回あたり削減時間を用いて、月間削減時間を試算してください。
```

### 9.2 指摘一覧

| 優先度 | 指摘 | 理由 | 修正案 | 根拠 |
| --- | --- | --- | --- | --- |
| 高 | 効果が定性的 | 投資判断が難しい | 削減時間を試算する | 過去レビュー方針 |
| 高 | 運用責任者が不明 | 導入後に形骸化する可能性がある | 管理者と更新フローを明記する | 過去レビュー方針 |
| 中 | セキュリティ説明が不足 | 生成AI利用時の懸念が残る | 入力禁止情報・ログ管理を追記する | 基本レビュー観点 |

### 9.3 適用した過去レビュー方針

```text
今回適用したレビュー方針：

1. 業務効率化提案では効果を定量化する
2. 生成AI提案では入力禁止情報とログ保存方針を記載する
3. 導入提案では運用責任者と更新フローを明記する
```

### 9.4 レビュワーへの確認事項

```text
AIだけでは判断できない点：

1. 今回の資料に定量効果を必須とするか
2. 顧客提出用か、社内確認用か
3. セキュリティ観点をどのレベルまで求めるか
4. 今回の判断を次回以降も標準方針にするか
```

### 9.5 学習カード候補

```text
タイトル：
生成AI提案におけるセキュリティ説明のレビュー方針

判断基準：
生成AIを利用する提案では、入力禁止情報、ログ保存方針、アクセス権限、利用者への注意喚起を記載することが望ましい。

推奨レビューコメント：
生成AI利用時の懸念を減らすため、入力してはいけない情報、ログ保存の有無、利用者権限、注意喚起の方法を追記してください。

適用条件：
生成AI、AIチャット、RAG、FAQボット、議事録AIなどの導入提案

除外条件：
社内検討前のアイディアメモ、技術検証のみの資料
```

## 11. MVPスコープ

### MVPで実現すること

MVPでは、以下に絞る。

```text
顧客提案書・社内稟議書を対象に、
AIが一次レビューし、
レビュワーが修正・承認し、
その判断をレビュー方針カードとして保存し、
次回の類似資料レビューに反映する。
```

### MVP機能一覧

| 優先度 | 機能 | 内容 |
| --- | --- | --- |
| 1 | 資料アップロード | PDF / PPT / テキストをアップロード |
| 2 | AI一次レビュー | 指摘事項、リスク、修正案を生成 |
| 3 | レビュワー修正 | AIレビュー結果を人間が編集 |
| 4 | 最終レビュー承認 | 人間がレビュー結果を確定 |
| 5 | 学習カード生成 | 修正差分からレビュー方針を生成 |
| 6 | 学習カード承認 | 次回以降に使うか人間が判断 |
| 7 | ベクトル保存 | 承認済みカードを保存 |
| 8 | 類似方針検索 | 次回レビュー時に過去方針を参照 |

## 12. 差別化ポイント

### 通常のAIレビューとの違い

通常のAIレビューは、資料を入れてAIがレビューして終わる。

```text
資料を入れる
↓
AIがレビューする
↓
終わり
```

Review Memory Agentは、レビュー後に人間の判断を蓄積し、次回に活かす。

```text
資料を入れる
↓
AIが一次レビューする
↓
レビュワーが修正・承認する
↓
判断をレビュー方針化する
↓
次回の類似レビューで参照する
↓
レビューするほど組織の判断基準が育つ
```

## 13. 最終メッセージ

### コンセプト文

Review Memory Agentは、レビュー判断の属人化と過去レビューの未活用を解消するAIエージェントである。

レビュイーが資料をアップロードすると、AIが基本的な一次レビューを実施する。その後、レビュワーがAIのレビュー結果を確認・修正・承認し、その判断をレビュー方針カードとして蓄積する。

次に類似資料がレビューされた際には、過去のレビュー方針を参照することで、組織固有の判断基準を踏まえた一貫性のあるレビューを支援する。

本プロダクトは、AIにレビュー判断を丸投げするのではなく、Human in the Loopによって人間の判断を組織知に変換する。レビューするたびに、組織のレビュー品質が継続的に高まるレビュー基盤を目指す。
